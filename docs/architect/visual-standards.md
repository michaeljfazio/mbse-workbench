## 2026-05-19 — Dim-17 edge-editing affordance conventions (walk-36 prereq)

Primary sources: OMG SysML v2 formal spec (`formal/2026-03-02`; cited as **SysML v2 §X**), OMG SysML 1.5 (`formal/2017-05-01`; cited as **SysML 1.5 §X**), OMG UML 2.5.1 (`formal/2017-12-05`; cited as **UML 2.5.1 §X**), and React Flow v12 API docs. Source priority per `README.md`.

### §A — Endpoint reconnect

Neither SysML v2 nor SysML 1.5 normatively prescribes a gesture for reconnecting an edge endpoint to a different target node; graphical-authoring UX is explicitly out of scope for both specs (SysML v2 §8.1 states the notation grammar defines rendered output, not tooling interactions). The closest normative anchor is UML 2.5.1 §7.8.7 (redefine associations), which describes the *semantic* result of changing an end-type but not the gesture. Vendor UX has converged on a grip-handle drag: **Cameo/MagicDraw** exposes a "Manipulate" mode (right-click → *Manipulate Edges*) and then drag either terminus handle to a new compatible target ([MagicDraw 2024x — Editing diagram elements](https://docs.nomagic.com/spaces/MD2024x/pages/136710445/Creating+diagrams)); **Eclipse Papyrus** renders small circular grip anchors at each edge end when the edge is selected, and the user drags a grip to reattach ([Papyrus User Guide — Diagram palette](https://wiki.eclipse.org/Papyrus_User_Guide)). React Flow v12 exposes `reconnectable` / `onReconnect` / `onReconnectStart` / `onReconnectEnd` props on `<ReactFlow>` and per-edge — no custom code is required to enable the UX; the workbench currently does not set `reconnectable`, so endpoint drag is absent ([React Flow v12 API — Edge reconnecting](https://reactflow.dev/api-reference/types/edge#reconnectable)).

### §B — Waypoints (routing bend points)

SysML v2 §8.2.3 and SysML 1.5 §8.2.2 define edge *paths* only as abstract graphical paths between source and target nodes; neither spec normatively defines waypoints or bend-point data structures. UML 2.5.1 §10.5.4 (DiagramInterchangeModel) specifies that a `DiagramElement` may carry an ordered sequence of `Point` waypoints in the DI metamodel, making waypoints a **standard DI concern** rather than a purely tool-private extension ([UML 2.5.1 §10.5.4 DiagramInterchangeModel](https://www.omg.org/spec/UML/2.5.1/PDF)). In practice, waypoint add/remove is a tool-UX feature: **Cameo/MagicDraw** exposes *Insert Break Point* and *Remove Break Point* via edge right-click context menu, plus "Smart Manipulators" that appear as blue dots along a selected edge ([Cameo SysML Plugin 2024x R2 — Manipulating edges](https://docs.nomagic.com/spaces/SYSMLP2024xR2/)). React Flow v12 does not ship a built-in waypoint add/remove API; waypoints must be implemented by storing an array of intermediate points in edge data and interpolating through them via a custom `EdgePath` calculation ([React Flow v12 — Custom Edges](https://reactflow.dev/learn/customization/custom-edges)).

### §C — Routing styles

SysML v2 §8.2.3 and SysML 1.5 §8.2.2 define edge paths as sequences of path segments but **do not mandate a specific routing algorithm** — routing style (straight, orthogonal step, bezier) is left to the tool. UML 2.5.1 Diagram Interchange §10.5.4 similarly leaves routing to tool discretion. The widely-adopted MBSE tool convention is: **orthogonal (rectilinear)** routing as the default for all structured-viewpoint edges (Cameo, Capella, SysON all default orthogonal for IBD `ConnectionUsage`, BDD associations, and Activity control flow); **smooth bezier** as an alternative where topology flexibility is needed; **straight** for simple two-node edges (Cameo optional). React Flow v12 ships five built-in edge types as path helpers: `default` (bezier, via `getBezierPath`), `straight` (via `getStraightPath`), `step` (right-angle step, via `getSmoothStepPath` with `borderRadius=0`), `smoothstep` (right-angle with rounded corners, via `getSmoothStepPath`), and `simplebezier` (via `getSimpleBezierPath`) ([React Flow v12 API — Edge types](https://reactflow.dev/api-reference/types/edge#type)). The workbench today hard-codes routing per edge kind (`getBezierPath` for IBD/Activity/State Machine; `getSmoothStepPath` for BDD/Requirements/UseCase/Package edges) — no per-edge routing-style picker is wired.

### §D — Label drag/placement

SysML v2 §8.2.3.13 (`connection-label`) places the edge name/type label at the **mid-edge** position by default; §8.2.3.13 also specifies `end-label` productions (rolename, multiplicity) placed **near the respective endpoint**. SysML 1.5 §8.2.2 Table 8.4 footnotes state that name labels ride the midpoint of the graphical path, with end-labels adjacent to the source/target anchor. Neither v1 nor v2 normatively specifies *manual label drag* — that is a tool-UX feature ([SysML v2 §8.2.3.13](https://www.omg.org/spec/SysML/2.0/); [SysML 1.5 §8.2.2 / Table 8.4](https://www.omg.org/spec/SysML/1.5/PDF)). Cameo/MagicDraw supports dragging the mid-edge label to any position along the edge and snapping it back to midpoint via "Reset Label Position." The workbench today renders all mid-edge labels at the `(labelX, labelY)` coordinate returned by `getBezierPath` / `getSmoothStepPath` and uses `pointerEvents: 'none'` — labels are therefore not draggable in the current implementation.

### §E — Edge style selection (line type and color semantics)

Line style is **load-bearing semantics** in SysML for several edge kinds: SysML 1.5 Table 8.4 specifies dependency edges as **dashed line** with open arrowhead; derive-requirement (`«deriveReqt»`) as dashed per §16.3.2.3; generalization as **solid** with hollow triangle; composition/aggregation as **solid** with diamond. SysML v2 §7.13 Table 11 carries these forward: `dependency` remains dashed; `FeatureTyping`/`Subclassification` (generalizations) remain solid; `ConnectionUsage` (IBD connector) is solid-no-arrowhead. Color is **not normatively semantic** in any SysML or UML spec — neither SysML v2 §7.13 nor SysML 1.5 Table 8.4 assigns meaning to hue. Color becomes **semantically loaded by tool convention** in two scenarios: (a) Cameo's impact-analysis highlighting where affected elements are colored orange/red, and (b) per-stereotype coloring in Capella's Arcadia layers ([Cameo SysML Plugin 2024x R2 — Impact analysis](https://docs.nomagic.com/spaces/SYSMLP2024xR2/); [SysML v2 §7.13 / Table 11](https://www.omg.org/spec/SysML/2.0/)). The workbench currently selects stroke color by `selected` state (primary vs foreground) with no semantic color variation per edge kind — consistent with the spec but does not yet support a per-edge style picker.

### §F — Cross-viewpoint routing variability

No SysML or UML spec section mandates routing style per diagram type; the following are **convergent vendor conventions** observed across Cameo, SysON, and Capella: (a) **IBD `ConnectionUsage`** — conventionally **orthogonal** (rectilinear with right-angle bends) because ports are fixed to block edges and orthogonal routing avoids diagonal crossing, yet the workbench renders bezier today; (b) **BDD associations/composition/generalization** — conventionally **orthogonal or straight**, depending on whether the diagram has a strict top-down hierarchy (generalization straight is common) or free-form layout (orthogonal step common); (c) **Activity control-flow edges** — conventionally **orthogonal** in swimlane layouts (tools such as Cameo and SysON default to rectilinear activity routing); (d) **State machine transitions** — conventionally **bezier** or **straight**, as self-loops and composite-state transitions require curve freedom; (e) **Requirement trace edges** — conventionally **bezier** or **orthogonal step** (no vendor consensus; the workbench uses `getSmoothStepPath`). No primary source mandates these routing styles; all citations for this subsection are vendor convention only: ([Cameo SysML Plugin 2024x R2 — Routing options](https://docs.nomagic.com/spaces/SYSMLP2024xR2/); [SysON user manual — Diagrams](https://doc.mbse-syson.org/syson/v2025.8.0/user-manual/features/diagrams.html)).
